## 🔗 关键深度主题

### 1. 可信度分层体系（5/20）

| 层级 | 内容类型 | 可信度 |
|:----:|----------|:------:|
| 🏅 最高 | 系统规则 | 最高 |
| ✅ 高 | 链上记录 / 工具返回结果 | 高 |
| 📄 中 | 文档 / 知识库 | 中（取决于版本和时效） |
| 👤 低-中 | 用户输入 | 低-中（需签名确认） |
| ⚠️ 低 | dApp 页面内容 | 仅作为声称 |
| ❌ 最低 | Memory / 历史记录 | 不能用于授权判断 |

### 2. Agent 铁律（5/21）

- Agent **绝不**接触私钥 / 助记词
- 签名和转账必须有**人工确认**
- Agent 的核心是被约束的执行循环，不是自由意志
- 只读操作可自动执行；高风险写入须 policy → simulation → 用户确认
- 状态管理应**外置**，不在模型上下文中（方便恢复、审计、追责）

### 3. Web3 概念的 AI 类比（5/21）

| Web3 概念 | AI 类比 |
|-----------|---------|
| 区块链网络 | 由无数节点共同维护的 append-only 状态数据库 |
| 钱包 | API Key + 公开 userId |
| 签名 | 私钥给消息"盖章"，加密可验证 |
| Gas | API 调用的 token 费用 |
| 智能合约 | 部署在链上的 serverless function（不可变） |
| 区块浏览器 | 链上的 Kibana / Grafana 日志面板 |
| EOA / Smart Account / Multisig | 个人账号 / 带 MFA 的企业账号 / N 人签名 |

### 4. Context Spec 元信息设计（5/20）

每个上下文字段需携带：`source`、`timestamp`、`freshness`、`trust_level`、`permission_scope`、`chain_id`、`block_number`、`verification_method`、`can_be_used_as_fact`、`expiry`
