## 1. 为什么要学这个

普通 AI 应用的上下文来自文档、聊天历史、数据库和网页。但 AI × Web3 多了一层：**链上状态持续变化**，而且很多状态和资产、权限、交易执行直接相关。

**如果 Agent 不知道：**
- 当前 chain id
- 合约地址
- ABI
- 用户授权
- 交易历史
- 数据更新时间

它就可能给出错误建议，甚至生成危险交易。

**一句话：** 链感知上下文的核心，是把链上事实变成模型可读、可引用、可验证的上下文。

---

## 2. 第一性原理

三个基本原则：

1. **模型不能凭记忆判断链上事实** → 链上事实必须从工具和索引层读取。模型知道"Uniswap 是 DEX"没用，真正执行时需要知道具体网络、具体合约、具体池子、当前价格、余额、allowance、滑点。

2. **链上状态有时间性** → 同一个地址的余额、授权和仓位随区块变化。上下文必须带来源：合约地址、区块号、交易哈希、explorer 链接。

3. **上下文要区分事实和解释** → 工具返回事实，模型负责解释。不要把模型猜测当事实。

## 3. 知识节点

### On-chain Data ⭐ 初级
链上可直接验证的数据：余额、交易、日志、合约状态、区块信息。
- 来源：RPC、区块浏览器、索引器、协议 API
- Agent 读取时**至少要带**：chain id、block number、contract address、method、返回值、读取时间
- **坑：** 缺少这些字段，模型容易把不同链、不同时间、不同合约的数据混在一起

### Contract Docs ⭐ 初级
帮助模型理解合约的设计意图、参数含义、权限边界。
- ABI 只能告诉函数签名，不能解释业务语义（比如 `execute(bytes)` 可能是普通执行，也可能是高权限入口）
- 需要 NatSpec、README、审计报告、部署说明补足语义
- **文档可能过期** → Agent 仍需用链上数据验证：合约地址、版本、owner、proxy implementation

### ABI / Event ⭐⭐ 中级
Agent 理解合约可调用能力和历史行为的核心结构。
- ABI → 如何编码调用、解码返回值、解析事件
- Event → 业务日志（Transfer、Swap、Deposit、VoteCast）
- **⚠️ 能调用 ≠ 应该调用** → 写交易前还需：权限、余额、allowance、slippage、simulation、policy 检查

### Transaction History ⭐⭐ 中级
理解用户或合约过去的行为。
- 用途：判断是否已授权、策略是否执行过、地址是否和高风险合约交互过
- **必须保留：** tx hash、block number、from、to、method、value、token transfers、logs
- 模型可总结，但证据必须能回到链上

### Explorer Context ⭐ 初级
区块浏览器提供的可视化链上证据。
- 交易是否成功、合约是否验证、源码是否公开、事件是否发出
- **在 AI 产品里，给 explorer link 比只给一句"交易成功"更可靠** → 用户可以自己打开核验

### Indexing Context ⭐⭐ 中级
把链上事件整理成面向产品查询的数据。
- Agent 需要的是"用户最近30天有哪些 DeFi 操作"，不是"某个区块里有什么"
- **必须带时间戳和同步状态** → 落后 500 个区块的索引结果不应该被当成当前事实

### Citation ⭐ 初级
让模型回答能回到具体链上证据或文档来源。
- 可引用：交易哈希、区块号、合约地址、event log、explorer 链接、文档 URL、审计报告章节
- **没有 citation 的链上解释 → 只是观点；带 citation → 可验证、可追责**

## 4. 在 AI × Web3 中的位置

```
                    ┌─────────────────────┐
                    │   Agent Wallet      │  ← 权限与控制
                    │  Agent Workflow     │  ← 工作流
                    │  Web3 Tool Use      │  ← 工具调用
                    └─────────┬───────────┘
                              │ 建立在可靠上下文上
                    ┌─────────▼───────────┐
                    │ Chain-aware Context │  ← 输入层（这章）
                    └─────────┬───────────┘
                              │ 读取链上事实
                    ┌─────────▼───────────┐
                    │  RPC / Indexer /    │
                    │  Explorer / API     │  ← 数据来源
                    └─────────────────────┘
```

Chain-aware Context 是所有链上 Agent 的**输入层**。之前学的 Agent Wallet 决定"能做什么"，而这章决定"知道什么"。

一个好的链感知上下文包应该包含：
1. 用户目标
2. 当前 chain id 和网络名称
3. 用户地址和余额
4. 相关合约地址、ABI、文档和风险提示
5. 最近交易和授权
6. 索引数据更新时间
7. 每条关键结论的 citation

---

## 5. 最小实践

**给一笔交易做上下文包：**
1. 找一笔公开交易哈希
2. 收集：chain id、block number、from、to、method、value、token transfers、logs
3. 找到合约 ABI 或 verified source
4. 写一段模型可读的上下文，每个关键结论附上交易哈希或 explorer 链接
5. 标出哪些是链上事实，哪些是你的解释

| 之前学的 | 和 Chain-aware Context 的关系 |
|---------|------------------------------|
| **Wallet** | Wallet 学了 EOA、签名、Gas — 这些是链上交互的基础，Chain-aware Context 负责在交互前提供准确的链上状态 |

## 7. 今日收获

### 新知识
- 🧠 Chain-aware Context = AI × Web3 的输入层，比工具调用和工作流更基础
- ⏱ 链上状态有时间性 — 上下文必须带区块号或时间戳
- 🔗 Citation 是链上 Agent 的可信基础设施，"交易成功"不如 explorer link
- ⚠️ ABI 能调用 ≠ 应该调用 — 写交易前需要多层检查

### 新技能
- 理解了如何构建一个结构化的链感知上下文包（7 个要素）
- 看懂了 Agent Wallet 和 Chain-aware Context 的分工：权限 vs 信息

### 关键洞察
> **之前做 Approval Tracker 时觉得是在解决授权可视化的 UX 问题，现在知道它本质上是在做链感知上下文的索引层。** 这个概念框架能让 MVP 的架构更清晰：扫描层（Indexed Context）→ 分析层（Model Interpretation）→ 展示层（Citation-backed UX）。
