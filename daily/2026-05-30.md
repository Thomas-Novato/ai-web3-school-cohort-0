## 1. 今日核心概念

**Chain-aware Context**，链感知上下文，指的是 AI 在回答、判断或执行链上动作之前，必须先看见正确的链上事实，包括：

- 当前链和 chain id
- 用户地址
- 合约地址、ABI
- 交易历史、Event logs
- 余额、allowance 授权
- 区块号、数据来源、数据更新时间
- explorer 链接或其他 citation

**核心目标**：让 AI 基于可验证的链上事实行动，而不是靠猜测。

> 普通 AI 上下文来自文档、聊天记录、数据库。AI × Web3 多了一层：链上状态持续变化，直接关系到资产、授权和交易安全。Agent 不知道当前网络、合约、ABI、授权、余额、数据更新时间 → 可能给出错误建议甚至构造危险交易。

---

## 2. 第一性原理

**模型不能凭语言记忆判断链上事实。** 链上事实必须从工具、RPC、索引器或区块浏览器读取。

模型知道 "Uniswap 是 DEX" 没有实际执行价值。真正执行一笔交易时，Agent 需要知道：
- 具体在哪条链
- 使用哪个合约
- 调用哪个函数
- 当前价格是多少
- 用户余额是否足够
- allowance 是否足够
- 滑点是否合理
- 交易模拟是否通过
- 数据来自哪个区块
- 结果是否可以追溯验证

**→ 链感知上下文不是"背景知识"，而是 AI × Web3 Agent 的输入安全层。**

---

## 3. 为什么 Chain-aware Context 很重要？

传统 AI 说错话 = 信息错误；链上 AI 说错话 = **真实资产损失**。

**可能的风险：**
- 资产转到错误地址
- 调用错误合约
- 给恶意 spender 授权
- 在错误网络执行交易
- 基于过期价格做 swap
- 重复执行已完成的操作
- 忽略合约升级、权限变更、风险事件

**链感知上下文的价值：** 把链上事实变成模型可读、可引用、可验证的上下文。

---

## 4. 链感知上下文的三个原则

### 4.1 上下文要有时间性
链上状态不是静态的，余额、授权、仓位都随区块变化。
Agent 不能只说"当前余额是多少"，要记录：读取时间、block number、chain id、数据来源、查询方法。
否则模型可能把旧数据当成当前事实。

### 4.2 上下文要有来源
每条关键结论最好回到证据：合约地址、交易哈希、区块号、event log、explorer link、文档 URL、审计报告章节、ABI 来源。

> **没有 citation 的链上解释 = 观点。带 citation 的解释 = 可验证、可追责。**

### 4.3 上下文要区分事实和解释
工具返回的是**事实**，模型负责**解释**。

- **链上事实：** 某地址在某个区块对某 spender 的 USDC allowance 是 100000000
- **模型解释：** 代表该 spender 仍有权限转走最多 100 USDC，需结合 spender 身份判断风险

事实和解释必须分开，模型不能把自己的推测包装成链上事实。

## 5. 关键知识节点

### 5.1 On-chain Data ⭐ 初级
链上可直接验证的数据：余额、交易、区块信息、合约状态、Event logs、Token transfers、Allowance、NFT owner。

- **来源：** RPC、区块浏览器、索引器、协议 API
- **Agent 读取时必须带：** chain id、block number、contract address、method、返回值、读取时间
- **坑：** 缺少这些字段，模型容易把不同链、不同时间、不同合约的数据混在一起

### 5.2 Contract Docs ⭐ 初级
帮助模型理解合约的设计意图、参数含义、权限边界和使用方式。

- ABI 只告诉函数签名，不告诉业务语义。例如 `execute(bytes calldata data)` 可能是普通入口也可能是高权限入口
- 还需参考：README、NatSpec、审计报告、部署说明、官方文档、治理说明
- ⚠️ **文档可能过期** → 读取后仍需用链上数据验证：合约地址、版本、proxy implementation、owner、事件是否匹配

### 5.3 ABI / Event ⭐⭐ 中级
Agent 理解合约可调用能力和历史行为的核心结构。

- **ABI →** 编码函数调用、解码返回值、识别函数签名、解析合约可调用方法
- **Event →** 链上业务日志（Transfer、Approval、Swap、Deposit、Withdraw、VoteCast）
- **⚠️ 能调用 ≠ 应该调用** → 写交易前仍需检查：权限、余额、allowance、滑点、交易模拟、合约风险、用户确认、policy 限制

**相关 topic：** 智能合约（Smart Contract）、索引（Indexing）

### 5.4 Transaction History ⭐⭐ 中级
帮助 Agent 理解用户或合约过去的行为。

- **可判断：** 是否已授权、策略是否执行过、是否交互过高风险合约、合约最近是否升级过、是否存在异常转账
- **必须保留：** tx hash、block number、from、to、method、value、token transfers、logs、status
- 模型可总结交易历史，但证据必须能回到链上

### 5.5 Explorer Context ⭐ 初级
区块浏览器提供的可视化链上证据。

- 交易是否成功、合约是否验证、源码是否公开、token transfer 是否发生、event 是否发出
- **在 AI 产品里，给 explorer link 比只说"交易成功"更可靠** → 用户可以自己核验，开发者能排查问题

### 5.6 Indexing Context ⭐⭐ 中级
把链上事件整理成产品可查询的数据。

- Agent 通常需要的不只是"某个区块里有什么"，而是"用户最近 30 天有哪些 DeFi 操作""这个池子的 TVL 变化如何""某个 Agent 做过哪些支付"
- **必须带：** 同步到哪个区块、数据更新时间、数据来源、查询条件、是否存在延迟
- 落后 500 个区块的索引结果，不应该被 Agent 当成当前事实

### 5.7 Citation ⭐ 初级
让模型回答能回到具体链上证据或文档来源。

- **可引用：** 交易哈希、区块号、合约地址、event log、explorer 链接、文档 URL、审计报告章节、GitHub commit、subgraph 查询结果
- **没有 citation 的链上解释 → 只是模型观点；带 citation → 才具备可验证性**

## 6. 在 AI × Web3 中的位置

```
                    ┌───────────────────────────┐
                    │ Agent Wallet              │  ← 权限与控制
                    │ Agent Workflow            │  ← 工作流
                    │ Web3 Tool Use             │  ← 工具调用
                    └───────────┬───────────────┘
                                │ 建立在可靠上下文上
                    ┌───────────▼───────────────┐
                    │  Chain-aware Context      │  ← 输入层（这章）
                    └───────────┬───────────────┘
                                │ 读取链上事实
                    ┌───────────▼───────────────┐
                    │ RPC / Indexer / Explorer  │  ← 数据来源
                    │ / Protocol API            │
                    └───────────────────────────┘
```

Chain-aware Context 是所有链上 Agent 的**输入层**。没有这层，Web3 Tool Use、Agent Workflow、Agent Wallet、自动交易、授权管理、风险分析、链上数据问答、DeFi 策略执行都会不可靠。

### 一个好的链感知上下文包包含：
1. 用户目标
2. 当前 chain id 和网络名称
3. 用户地址和余额
4. 相关合约地址、ABI、文档、风险提示
5. 最近交易和授权状态
6. 索引数据更新时间
7. 每条关键结论的 citation

## 7. 今日自测

| # | 问题 | 答案 |
|---|------|------|
| Q1 | 什么是 Chain-aware Context？ | 让 AI 在行动前读取正确的链、地址、合约、交易、授权、余额和数据来源，基于可验证的链上事实行动 |
| Q2 | 为什么普通上下文不够？ | 链上状态随区块变化，常规上下文（聊天记录、文档）不包含可验证的链上事实 |
| Q3 | 链上数据读取至少保留哪些字段？ | chain id、block number、contract address、method、返回值、读取时间、数据来源 |
| Q4 | ABI 能完全解释合约吗？ | 不能。ABI 只告诉函数签名和参数类型，缺少业务语义、权限边界和风险判断 |
| Q5 | Event 为什么重要？ | 链上业务日志，是索引器和 Agent 理解链上历史行为的基础 |
| Q6 | Transaction History 能判断什么？ | 授权状态、策略执行、合约风险、合约升级、交易是否成功、异常转账 |
| Q7 | Explorer Context 的价值？ | 提供可检查入口，用户可自行核验交易状态和合约信息 |
| Q8 | Indexing Context 为什么要带同步状态？ | 索引可能滞后，落后 500 区块的结果不能当当前事实 |
| Q9 | 什么是 Citation？ | 模型结论的源头引用（tx hash、block number等），让回答可验证、可追责 |
| Q10 | 和 Agent Wallet 的关系？ | Chain-aware Context 是 Agent Wallet 安全执行的前置条件——不知道链状态，就不能安全控制资产 |

## 8. 与之前的连接

| 之前学的 | 和 Chain-aware Context 的关系 |
|---------|------------------------------|
| **Wallet** | Wallet 学了 EOA、签名、Gas — 这些是链上交互的基础，Chain-aware Context 负责在交互前提供准确的链上状态 |
| **Agent Wallet** | Agent Wallet 讨论授权粒度（能做什么），Chain-aware Context 讨论信息完整性（知道什么）— 两者缺一不可 |
| **Network** | Network 学了下交易链路，Chain-aware Context 是把这条链路变成模型可读的上下文 |

**关键洞察 — 和 Approval Tracker MVP 的关系：**
> 之前做 Approval Tracker 时觉得是在解决授权可视化的 UX 问题，现在知道它本质上是在做链感知上下文的索引层。这个概念框架能让 MVP 的架构更清晰：扫描层（Indexed Context）→ 分析层（Model Interpretation）→ 展示层（Citation-backed UX）。

## 9. 今日收获

### 新知识
- 🧠 Chain-aware Context = AI × Web3 的输入层，比工具调用和工作流更基础
- ⏱ 链上状态有时间性 — 上下文必须带区块号或时间戳
- 🔗 Citation 是链上 Agent 的可信基础设施，"交易成功"不如 explorer link
- ⚠️ ABI 能调用 ≠ 应该调用 — 写交易前需要多层安全检查
- 🛡 链上 Agent 不应说"我觉得这个地址安全"，而应说"根据某链、某区块、某合约、某笔交易和某 explorer 证据，我判断……"

### 新技能
- 理解了如何构建一个结构化的链感知上下文包（7 个要素）
- 看懂了 Agent Wallet 和 Chain-aware Context 的分工：权限 vs 信息
- 掌握了链上场景中事实与解释的分离原则
