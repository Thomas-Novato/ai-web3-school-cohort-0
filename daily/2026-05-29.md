## 1. 概念学习：Agent Wallet

### 核心判断

> **AI 可以执行任务，但不能无边界地控制资产。**

真正靠谱的 AI × Web3 产品，不是让 Agent 直接拿走用户的钱包权限，而是让它只能在用户预先允许的范围内行动：

- 能做什么
- 能花多少
- 能调用谁
- 什么时候失效
- 怎么撤销

这些都必须被明确写成规则。

### 从概念到现实问题

顺着 Agent Wallet 继续往下想，发现一个更基础的现实问题：

> 很多用户其实根本不知道自己曾经给哪些合约授权过，也不知道哪些授权现在仍然有效。

→ 这就是做 **Approval Tracker** 的原因。

## 2. 项目设计：Approval Tracker MVP

### 项目定位

**不是一个"帮用户自动撤销授权"的完整钱包产品**，而是一个更适合黑客松 MVP 的安全扫描工具：

> 先帮用户看清楚：我给过谁授权？现在还有多少 allowance？风险大不大？交易记录在哪里？

**不碰私钥，不请求签名，不提交交易，只做可验证数据扫描和风险解释。**

### 与 Agent Wallet 的模块关系

| 模块 | 职责 |
|------|------|
| **Agent Wallet** | 关注如何授权 |
| **Approval Tracker** | ⬅ **关注已有授权是否安全** |
| Policy | 定义边界 |
| Guard | 执行前拦截 |
| Revocation | 权限收回 |

Approval Tracker 可以理解成 Agent Wallet 安全链路里的一个**基础模块**，也是一份 Agent Wallet 安全面板原型。

### 双入口设计

```
                    ┌─ CLI 命令行 (approval-tracker scan 0x... --format table/json)
  shared scanner ──┤
                    └─ Telegram Bot (/scan 0x... → 中文风险摘要)
```

同一套扫描核心，两种展示入口，CLI 和 Bot 不会变成两个半成品。

## 3. 技术方案

### 关键修正

没有直接用旧思路的 `tokenapproval` 接口，而是：

1. 通过 **Etherscan V2 logs.getLogs** 查询 `Approval(address,address,uint256)` 事件日志
2. 再用 **eth_call** 调用 ERC-20 合约的 `allowance(owner, spender)`，读取当前有效授权

> 历史上出现过 Approval 事件 ≠ 现在仍然有授权。只看历史 logs 不够，必须再查当前 allowance。

### 核心数据流

```
  用户输入 EVM 地址
       │
       ▼
  ① 校验地址格式
       │
       ▼
  ② 分别扫描 Ethereum 主网 + Sepolia
       │
       ▼
  ③ logs.getLogs 查 Approval 事件
     (topic0 = 事件签名, topic1 = owner 地址)
       │
       ▼
  ④ 解析 log → token 合约 / spender / amount / tx hash / block / timestamp
       │
       ▼
  ⑤ 对每组 (token, owner, spender) 调用 allowance()
       │
       ▼
  ⑥ 过滤 allowance = 0 (已失效的授权)
       │
       ▼
  ⑦ 读取 ERC-20 symbol / name / decimals (失败则降级显示地址)
       │
       ▼
  ⑧ getsourcecode(spender) → 是否 verified + contract name
       │
       ▼
  ⑨ 本地标签库匹配 → Uniswap / Aave / OpenSea / 1inch ...
       │
       ▼
  ⑩ 生成风险等级 + 中文报告
```

### 风险评级规则

| 等级 | 条件 | 说明 |
|------|------|------|
| 🔴 **Critical** | 合约未验证 + 无限额度 | 源码不可见的合约拿了长期大额权限 |
| 🟠 **High** | 未知地址 + 无限额度 | 非常见协议 + 无限授权 |
| 🟡 **Medium** | 已知协议 + 无限额度 | 知名协议但无限授权仍扩大攻击面 |
| 🟢 **Low** | 有限额度 + 信息明确 | 无明显异常 |

> 系统不会把"未验证合约"直接判恶意，而是作为风险因子之一。

## 4. 环境变量与安全边界

### 环境变量策略

| 变量 | 必需性 | 说明 |
|------|--------|------|
| `ETHERSCAN_API_KEY` | **必需** | 扫描链上数据的核心依赖 |
| `TELEGRAM_BOT_TOKEN` | 仅 Bot 入口必需 | CLI 模式不需要 |
| `OPENAI_API_KEY` / `OPENAI_BASE_URL` | **可选** | 无 LLM key 时规则模板兜底 |

**关键设计：** LLM 只能改写解释，不能新增事实。
所有事实必须来自 Etherscan API、链上调用结果或本地标签库。系统在无 LLM key 时仍然能跑通 Demo。

### 安全边界（红线清单）

| 不做 | 理由 |
|-----|------|
| ❌ 不请求私钥 / 助记词 | 永远不要 |
| ❌ 不要求用户签名 | MVP 只读扫描 |
| ❌ 不提交链上交易 | MVP 只读扫描 |
| ❌ 不自动 revoke | 高风险操作留给用户决策 |
| ❌ 不支持 NFT setApprovalForAll | 超出 ERC-20 范围 |
| ❌ 不支持 EIP-2612 Permit | 签名消息，MVP 不做 |
| ❌ 不支持定期报告 / 实时监控 | MVP 按需扫描 |

**MVP 只做：** ERC-20 Approval 扫描 → 当前 allowance 校验 → 风险评级 → 中文报告

> 黑客松项目最怕"看起来很完整，但每一块都不稳"。先把一个小闭环跑通。

## 5. 测试计划与展示思路

- EVM 地址校验 / Approval topic 编码解码 / allowance() calldata
- 无限额度识别 / 金额格式化 / 风险评级规则
- Etherscan 错误处理 / 空结果处理 / 限速和分页逻辑

**集成测试：**
- 用 fixture 模拟 Ethereum 和 Sepolia API 响应
- 跑通 scan → 报告生成完整流程
- 验证 CLI JSON 输出可解析、table 输出包含字段
- Mock bot 验证 /scan 能返回摘要和风险项

**手动验收：**
- 配置 `ETHERSCAN_API_KEY` 后扫描一个真实地址
- 配置 `TELEGRAM_BOT_TOKEN` 后启动 Bot
- Telegram 中发 `/scan 0x...` 收中文风险报告
- **无 LLM key 时仍然可以生成规则报告**
- **无 API key 时 README 有明确报错说明**

### 黑客松展示流程

```
Step 1 ── CLI 扫描真实地址
          approval-tracker scan 0x... --chains ethereum,sepolia --format table

Step 2 ── 展示风险表格
          → 哪些无限授权 / 哪些未知地址 / 哪些未验证合约

Step 3 ── 切到 Telegram Bot
          /scan 0x...

Step 4 ── Bot 返回中文摘要
          → 不只是开发者可用，普通用户也能理解

Step 5 ── 强调安全边界
          → 不替用户签名 / 不替用户交易
          → 只帮助用户看清楚已有权限
```

> 这跟 Agent Wallet 的理念一致：让自动化更有用，但不能让权限失控。

## 6. 与 Agent Wallet 的深层连接

今天最大的收获：**把 Agent Wallet 从抽象概念向真实产品推进了一步。**

| 维度 | Agent Wallet | Approval Tracker |
|------|-------------|-----------------|
| 讨论 | Agent 如何在有限权限内安全行动 | 用户能不能先看清已给权限 |
| 问题 | 如何授权才安全 | 已有授权是否还安全 |
| 角色 | 授权机制 | 安全面板原型 |

这两个问题连在一起，就是 **AI × Web3 安全体验的关键**：

真正有价值的 Web3 Agent，不应该只是帮用户"生成交易"，而应该帮助用户理解：

- ✅ 我授权了什么
- ✅ 谁还能花我的 token
- ✅ 当前风险在哪里
- ✅ 哪些操作应该自动执行
- ✅ 哪些操作必须人工确认
- ✅ 哪些权限应该及时撤销

> **不是让 AI 帮用户接管钱包，而是让用户重新看见并控制自己的权限。**

## 7. 今日收获

### 新知识

- Agent Wallet 的核心在于**受限权限下的安全执行**，不是让 Agent 直接控制资产
- ERC-20 Approval 的历史事件 ≠ 当前有效授权，必须 `allowance()` 确认
- 无限授权即使给知名协议也有风险（攻击面扩大）
- 风险评级不应把"未验证合约"直接判恶意，而是作为风险因子之一
- LLM 在安全产品中的角色边界：只能改写解释，不能新增事实

### 新技能

- 用 Etherscan V2 logs.getLogs 查询链上事件的完整流程
- ERC-20 `allowance()` eth_call 读取当前授权
- 构建本地协议标签库做风险识别
- MVP 设计：共享扫描核心 + 双入口（CLI + Telegram Bot）
- 明确安全边界：MVP 的"不做"清单和"只做"闭环

### 疑问

- 如何处理大量授权的分页（Etherscan API 单次限制）？
- 是否要支持其他 EVM 链（Polygon、Arbitrum、Optimism）？
