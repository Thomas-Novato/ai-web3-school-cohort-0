## 学习内容

### 为什么要学钱包

Web3 应用离不开钱包。用户连接钱包、签名消息、发送交易、批准 token、切换网络——这些动作都发生在**钱包边界**上。

关键洞察：**钱包连接成功 ≠ 用户理解了后续动作。** 真正重要的是用户是否知道：
- 自己在哪条链上
- 和哪个合约交互
- 授权了什么资产
- 要付多少 gas
- 失败后能不能恢复

> 💡 钱包是用户意图进入链上执行之前的**最后一道确认界面**

**三类钱包动作，风险完全不同：**
1. **连接钱包** — 应用读取地址和网络，不可动用资产
2. **签名消息** — 证明控制权，可能用于登录/授权/创建订单
3. **发送交易** — 改变链上状态，转账/授权 token/调用合约

### 第一性原理

| 传统互联网 | 钱包体系 |
|-----------|---------|
| 平台管理账号资料 | 控制权来自私钥或智能账户规则 |
| 可重置密码 | 私钥丢失 = 永久失去 |
| 应用假设拥有用户账户 | 应用只能请求连接和签名 |

三原则：
- **连接不是授权资产** — 读地址和发交易是两层完全不同的权限
- **签名必须可解释** — 用户需要知道签名目的，不是只看十六进制数据
- **网络是上下文** — 同一地址在不同链上可能有不同资产和状态

### 知识节点

#### 1. EOA（Externally Owned Account）— 初级
私钥控制的账户，可签名、发交易、付 gas、调合约。MetaMask 默认创建这类账户。
- ✅ 优点：简单、通用、生态兼容好
- ❌ 限制：私钥丢失难恢复，权限难细分，自动化弱，用户体验常被弹窗打断
- 🏷️ 关联：**账户抽象（Account Abstraction）**— 从私钥扩展到可编程规则

#### 2. Mnemonic（助记词）— 初级
一组单词，用来恢复钱包私钥。**不是普通验证码，任何应用都不该索要。**

> ⚠️ 钓鱼常见手法：伪装成"输入助记词恢复账户"
> 
> 任何网页、客服、AI Agent、表单或脚本要求你输入助记词 → 默认视为危险

产品正确的做法：让用户在钱包应用内部控制恢复，然后 dApp 请求连接。

#### 3. Transaction（交易）— 中级
对链上状态的正式请求，成功后通常不可撤回。

> ❌ 常见误区：点击按钮 = 交易发送成功
> 
> ✅ 实际流程：按钮发起请求 → 钱包确认 → 签名 → 广播 → 打包 → 执行

**前端需要展示的状态：**
| 状态 | 说明 |
|------|------|
| 等待确认 | 用户在钱包里弹窗确认 |
| 用户拒绝 | 签名被取消 |
| 已广播 | 等待区块确认 |
| 成功 | 交易上链 |
| 失败 / Revert | 交易回滚 |
| Pending | 长时间未确认，需重试或查看 explorer |

#### 4. Gas（燃料费）— 初级
链上执行成本。影响付费和交易是否及时处理。

产品不能只显示"确认交易"。至少让用户知道：
- 大概费用是多少
- 用什么资产支付
- 失败时是否仍消耗费用
- 是否可以重试

#### 5. Explorer（区块浏览器）— 初级
查看链上事实的窗口，但不是链本身。

**看一笔交易至少要检查：**
- Status（成功/失败）
- From / To（谁发起、调用了哪个地址）
- Method（调用的合约方法）
- Value（是否转了原生资产）
- Token Transfers（是否有 token 转移）
- Logs（合约发出的事件）
- Gas Used（实际消耗）

> ✅ AI 产品引用链上信息时，应给出 explorer 链接或交易哈希，让用户自己验证

### 在 AI × Web3 中的位置

AI Agent 进链上执行，最终会碰到**钱包边界**。

- AI 可以帮：解释交易、准备参数、检查风险、生成操作计划
- AI 不能：随意签名、管理权限

> 合理设计：**AI 做理解和辅助 → 钱包负责确认和授权 → 策略合约/智能账户负责执行约束**

这样用户既有 AI 的效率，也保留对资产和权限的控制。
