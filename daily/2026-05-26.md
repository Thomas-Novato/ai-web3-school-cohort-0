> 完成了 AI 基础模块中 MCP 章节的学习，重点理解了 MCP 作为 AI 与外部工具标准化连接协议的设计哲学和实践方法。

---

## 学习内容

### 什么是 MCP

MCP 是一套让 AI 与外部工具、资源和系统进行**标准化连接**的协议。其核心架构包括三个角色：

| 组件 | 职责 |
|------|------|
| **Model** | AI 模型本身，负责推理和决策 |
| **MCP Client** | 能力发现 + 工具调用编排 |
| **MCP Server** | 暴露资源和工具能力 |

Client 负责"发现"Server 暴露了哪些能力，并将这些能力呈现给 Model。

### MCP 的核心：能力边界管理

MCP 的重点不是"能否连接"，而是**能力边界管理**。Server 需要明确：

- 暴露哪些**资源**（read / write 权限）
- 哪些工具具有**副作用**（状态变更、链上交易）
- **参数 Schema** 如何设计（决定模型是否能正确调用）
- **权限**如何控制（模型能调用到什么程度）

> **关键洞察**：Tool Schema 决定模型是否能够正确调用工具，Permission 则决定模型能调用到什么程度。前者是能力问题，后者是安全边界问题。

### AI × Web3 中的 MCP 应用

MCP 可以作为 Agent 与链上工具之间的**接口层**：

- ✅ **只读操作**：读取区块链数据、查询合约信息（低风险，可自动执行）
- ⚠️ **写入操作**：生成交易草稿、签名提案（需要额外安全机制）
- ❌ **高风险**：直接转账、合约部署（必须经过多层审批）

> ⚠️ MCP 本身不是安全方案。最终权限控制仍然依赖：
> - **钱包授权** — 链上权限根源
> - **交易模拟** — 预执行验证
> - **用户签名** — 人工确认环节
> - **审计系统** — 事后追踪和问责

### 最小安全实践

安全的 MCP Server 应该从以下原则开始设计：

1. **从只读开始** — 先暴露读数据的能力
2. **逐步增加权限控制** — 需要时才开放写入
3. **用户确认机制** — 高风险操作必须有确认环节
4. **审计系统** — 所有操作可追踪和回溯

不要一开始就开放高风险写入能力。

### 与课程其他概念的关联

| 关联概念 | 说明 |
|----------|------|
| **Agent 章节** | MCP 是 Agent 的工具调用接口标准化方案 |
| **Context 章节** | MCP Server 返回的数据也需要携带 trust_level、source 等元信息 |
| **Web3 Tool Use** | MCP 可以为链上调用提供统一接口层 |
| **AI Security** | MCP 的权限控制和审计机制是 AI 安全的重要组成部分 |

## 今日收获

### 新知识
- MCP 的三层架构：Model → Client → Server
- 能力边界管理比连接能力更重要
- Tool Schema 决定调用正确性，Permission 决定调用深度
- MCP 本身不是安全方案，需要组合钱包授权、交易模拟、用户签名、审计系统

### 疑问
