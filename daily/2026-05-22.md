### 理论学习：MCP 与 Eval

今天学习了 AI 应用工程化的两个重要主题：**MCP（Model Context Protocol）** 和 **Eval（评估体系）**。

简单理解两者的关系：
- **MCP** 关注"模型如何连接外部世界"
- **Eval** 关注"连接之后系统表现是否可信"
- 在 AI × Web3 场景中，二者都服务于安全、可控、可验证的 Agent 系统

---

## MCP：模型连接外部能力的协议层

### 核心认知

MCP 试图把模型与外部工具、数据源、应用上下文之间的连接方式标准化。

它解决的不是"模型会不会更聪明"，而是 **"模型如何以可描述、可复用、可管理的方式使用外部能力"**。

可以把 MCP 理解成 AI 应用时代的"工具接口标准"。

### 为什么学 MCP

当 AI 应用真正进入工作流后，模型不可能只靠 prompt 里的文本完成任务。它需要：
- 读取文件、查询数据库
- 访问文档、调用 API
- 操作工具、获取用户上下文

如果每个应用都用自己的方式接工具，Agent 生态会非常碎片化。MCP 的价值在于把这层连接**抽象成协议**，让工具接入变得标准、清晰、可复用。

### 第一性原理

> 模型不应该直接拥有世界，而应该通过明确协议访问被授权的上下文和工具。

MCP 的关键不是"能接更多工具"，而是让工具接入变得：
- **可描述** — 能力清单清晰
- **可发现** — Client 能自动发现 Server 能力
- **可限制** — 权限边界明确
- **可复用** — 一次接入，多处使用
- **可审计** — 调用记录可追溯

---

### MCP 关键知识节点

#### 1. Server

MCP Server 是提供能力的一侧。它可以暴露：

| 类型 | 说明 | 示例 |
|------|------|------|
| **resources** | 资源 | 文件、文档、数据库记录 |
| **tools** | 工具 | 搜索、查询、API、创建 PR |
| **prompts** | 提示模板 | 可复用的工作流入口 |

Server 的设计重点是**边界**：

- 暴露哪些资源
- 哪些工具只读 / 哪些有副作用
- 参数 schema 是否清楚
- 错误如何返回
- 是否需要用户授权
- 日志和审计在哪里记录

> ⚠️ 一个不加限制的 MCP server，很容易把本地文件、内部 API 或高风险动作暴露给模型。

#### 2. Client

MCP Client 是连接模型和 MCP Server 的一侧（桌面应用、IDE、Agent runtime、聊天客户端）。

Client 负责：
- 发现 Server 能力
- 把工具信息交给模型
- 将工具调用请求转成协议消息
- 展示工具调用过程
- 处理用户确认和权限提示
- 会话隔离

判断 Client 是否可靠，重点看是否让用户知道：
- 当前连接了哪些 Server
- 模型能调用哪些工具
- 哪些调用需要确认
- 哪些工具有写入或外部副作用

#### 3. Tool Schema

Tool Schema 描述工具的名字、用途、参数、返回值和约束。

好的 Schema 不只是字段类型，还应该说明：
- 这个工具什么时候用
- 参数代表什么
- 哪些字段必填
- 是否会修改外部状态
- 失败时返回什么

> Tool Schema 本质上也是 prompt 的一部分。Schema 写得越模糊，模型越容易用错误参数填空。

#### 4. Permission（权限）

Permission 是 MCP 真正进入产品时**最容易被低估的问题**。

MCP 让工具连接更方便，但方便不等于安全。不同操作的风险完全不同：
- 读一个文档 ✅
- 查一个 issue ✅
- 创建一个 PR ⚠️
- 删除一个文件 ❌
- 发起一笔交易 ❌

权限至少要区分：

| 维度 | 说明 |
|------|------|
| 只读 vs 写入 | 是否修改外部状态 |
| 会话 vs 长期 | 授权有效期 |
| 用户确认 | 是否需要手工确认 |
| 敏感信息 | 能否访问敏感数据 |
| 副作用 | 是否有外部影响 |
| 可撤销 | 能否撤销授权 |

> MCP 可以描述能力，但真正的**授权、审计、隔离和风控**仍然要由系统实现。

## Eval：持续判断 AI 系统是否可靠

### 核心认知

Eval 是对 AI 系统进行**系统化测试和评分**的机制。

它不是只看"这次回答对不对"，而是让团队能够持续比较：
- 不同 prompt 的效果
- 不同模型的效果
- 不同检索策略的效果
- 不同工具调用策略的效果
- 系统是否出现 **regression**

> 没有 eval，AI 应用的改动很容易变成玄学。

---

### Eval 关键知识节点

#### 1. Harness（测试框架）

Harness 是运行 Eval 的框架。它负责：

- 喂样本
- 调用被测系统
- 收集输出
- 运行 grader
- 记录结果
- 生成报告

一个最小 Harness 至少需要：

| 组件 | 说明 |
|------|------|
| 输入样本 | 测试数据 |
| 期望输出或评分规则 | 评判标准 |
| 被测系统版本 | 可复现性 |
| 模型和参数配置 | 环境信息 |
| 运行日志 | 调试依据 |
| 结果报告 | 比较基线 |

> Harness 的核心价值是**可重复**。没有可重复运行的 eval，就很难比较差异。

#### 2. Golden Set（黄金测试集）

Golden Set 是一组被认真挑选和标注的测试样本。**不一定要很大**，早期 30-100 条高质量样本，往往比大量随便收集的问题更有用。

好的 Golden Set 包含：

| 类型 | 说明 |
|------|------|
| 常见正常问题 | 覆盖核心场景 |
| 边界问题 | 测试极限情况 |
| 容易误判的问题 | 突出薄弱环节 |
| 高风险问题 | 安全边界 |
| 历史 bug | 防 regression |
| 用户真实反馈 | 贴近实际使用 |

> 每修一个重要 bug，都应该考虑把它变成 regression 样本。

#### 3. LLM-as-Judge（模型评分）

LLM-as-Judge 是用模型来给模型输出评分。适合评估开放式答案：

- 摘要质量
- 回答是否完整
- 是否遵循格式
- 是否引用来源
- 风险提示是否充分

但它**不能**被神化。Judge 模型也会偏、会漏、会被输出风格影响。

更稳的做法：

| 方法 | 适用场景 |
|------|----------|
| 规则评分 | 可自动判断的字段 |
| LLM Judge | 开放式质量评估 |
| 人工抽检 | 高风险样本 |
| 定期校准 | 保持 judge 和人工一致 |

> LLM-as-Judge 是评估工具，不是最终真相。

#### 4. Regression（回归测试）

Regression 的目标是**防止旧问题复发**。

AI 应用很容易出现"修 A 坏 B"。一次 prompt 修改、一次模型升级、一次 retriever 调整，都可能影响很多旧场景。

实用流程：

1. 用户反馈一个错误
2. 复现并记录输入
3. 标注期望输出或拒答条件
4. 加入 Regression Set
5. 之后每次发布前重新运行

> Regression Set 的价值是把历史问题固定下来，防止系统反复踩同一个坑。

#### 5. Observability（可观测性）

Observability 是**线上观察系统行为**的能力。

Eval 多数发生在发布前，Observability 发生在真实使用中。

至少要记录：

| 数据 | 说明 |
|------|------|
| 输入类型和来源 | 用户从哪来 |
| 检索结果 | RAG 给模型喂了什么 |
| 工具调用 | 调用了哪些工具 |
| 模型输出 | 生成了什么 |
| 错误和重试 | 哪里失败了 |
| 用户反馈 | 用户满意吗 |
| 成本和延迟 | 系统性能 |

> 没有 observability，就不知道真实用户在哪里失败，也不知道该往 golden set 里补什么样本。

## 在 AI × Web3 中的位置

在 AI × Web3 系统里，MCP 和 Eval 都更重要，因为**错误可能影响资产、权限、治理判断和链上执行**。

### MCP 可以做的事
- 读取区块浏览器信息
- 查询合约文档
- 调用 RPC
- 生成交易草稿
- 读取项目 issue
- 分析治理提案

⚠️ MCP 本身**不是钱包安全方案**。它可以标准化工具入口，但不能替代账户权限、交易模拟、签名确认、Session Key 和审计日志。

### Eval 需要特别评估
- 交易解释是否准确
- 风险提示是否漏报
- 工具调用参数是否越界
- 是否能拒绝不确定请求
- 是否能识别 Prompt Injection
- 引用和来源是否可追溯
- 高风险动作是否要求 human check

### 比较稳的设计

```
MCP 负责工具发现和调用格式。
Eval 负责持续发现系统不可靠的场景。
Web3 账户系统负责最终权限和执行边界。
```

## 今日收获

### 🆕 新知识

1. **MCP 的本质**：标准化模型访问外部上下文和工具的方式
2. **Eval 的本质**：用可重复的测试判断 AI 系统是否可靠
3. **MCP Server/Client 架构**：Server 提供能力，Client 连接模型和 Server，边界设计是关键
4. **Tool Schema 的重要性**：写得越模糊，模型越容易用错误参数填空
5. **Permission 是 MCP 最被低估的问题**：方便不等于安全
6. **Golden Set 的 30-100 条原则**：质量远大于数量
7. **LLM-as-Judge 的双刃剑**：好用但有偏见，需要规则+人工配合
8. **Regression 的价值**：修 A 坏 B 是 AI 应用的常态
9. **Observability 是 Eval 的补充**：只有线上才知道真实失败点

### 🛠️ 最小实践方案

**只读 MCP Server：**
- `search_docs(query)` — 搜索本地项目文档
- `get_file(path)` — 读取白名单目录里的文件
- 约束：不能读白名单外路径、返回带来源路径、写日志、错误明确返回

**最小 Eval（30 条样本）：**
- 10 条正常问题
- 10 条边界/易混淆问题
- 5 条历史 bug
- 5 条恶意/注入样本

**每条样本定义：** 输入 → 期望行为 → 必须包含的信息 → 必须拒绝的情况 → 是否需要引用

### 💡 关键启示

> MCP 不能替代钱包安全，Eval 不能替代权限控制，高风险动作必须有最终的人类确认或账户系统约束。
