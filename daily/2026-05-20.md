今天通过 Codex 阅读了 **Context（上下文）** 相关内容，并结合 AI x Web3 Bridge 场景，深入思考了 6 个核心问题。

---

### 深入思考 6 问

#### Q1: 为什么更大的 Context Window 不一定等于更可靠的 AI 应用？

更大的 context window 只能装更多内容，不代表模型能完美理解、排序、判断重要性。

核心问题：
- 模型对长上下文的注意力分布不均匀（lost in the middle）
- 大量无关、过期、冗余信息会稀释关键信号的权重
- 上下文里如果有过期文档、无关日志、旧任务状态，会干扰模型判断
- 更大的窗口意味着更大的风险面——更多不可信内容可能混入

结论：Context 的质量大于 Context 的数量。要主动管理上下文的来源、时效和可信度。

---

#### Q2: 用户说这个地址是我的钱包，模型能不能直接当成事实？

不能。模型不能把用户的主观陈述直接当成事实，原因包括：

1. 安全风险 — 可能是盗取的地址、钓鱼地址
2. 身份冒用 — 这个地址可能属于别人
3. 人为失误 — 用户可能输错、复制错
4. 意图不明 — 用户可能只是想查询，不是要授权
5. 未验证 — 没有完成当前会话的钱包签名验证

原则：涉及身份和资产时，必须经过当前会话的签名和授权确认。口头陈述不等于事实。

---

#### Q3: 在钱包授权检查 Agent 里，哪些信息必须实时查询？哪些可以缓存？

必须实时查询的有：chain ID、当前区块号、Token 合约地址、Spender 地址、Approve 数量、用户当前 Allowance、用户余额、Spender 合约状态、Simulation/静态检查结果

可以缓存但要标注更新时间的有：已知可信 dApp 列表、Token 元数据、项目文档、历史安全报告

不能直接当事实的有：dApp 页面自己写的说明、用户的主观判断、历史 memory、社区评论、搜索结果摘要

---

#### Q4: dApp 页面上写着这是安全授权，为什么应该被标记为不可信外部内容？

因为 dApp 页面是外部来源，可能被篡改、误导或恶意设计。

原因拆解：
- 页面内容可以被前端修改、中间人攻击、DNS 劫持
- 恶意 dApp 可以伪造授权说明
- 页面声称和合约实际行为可能不一致（approve 钓鱼）

原则：它只能作为「页面声称」，不能作为安全事实。模型必须以链上数据、合约检查和 simulation 结果为准。

---

#### Q5: Memory 可以提升用户体验，但在 Web3 场景最大的风险是什么？

Memory 可以保存偏好和历史，但不能保存或延续授权。

关键风险：
- 授权具有时效性 — 过去的授权可能已被撤销
- 授权具有会话绑定 — 不能跨会话重用
- 资产和权限状态可能在变化
- Memory 中的历史记录可能导致过期授权幻觉

原则：涉及身份、资产、权限、隐私和外部操作时，必须重新绑定当前会话、当前用户和当前授权。

---

#### Q6: 如果知识库里有一篇旧版 SDK 文档，RAG 把它检索出来，会有什么问题？

RAG 不会自动保证内容正确。它只是把检索到的材料送进上下文。

问题链：
1. RAG 检索到旧版文档（因为语义相似）
2. 模型把旧版文档当成当前依据
3. 生成过期甚至危险的代码或建议
4. 用户基于错误信息操作造成资产损失

知识库管理原则：
- 知识库不是越多越好
- 每条知识应标注：来源、版本、更新时间、适用范围、可信度、是否废弃
- 定期清理过期文档

### 可信度分层体系

| 层级 | 内容类型 | 可信度 | 说明 |
|------|----------|--------|------|
| 最高 | 系统规则 | 最高 | 定义安全边界和行为约束 |
| 高 | 链上记录 | 高 | 客观事实，可验证 |
| 高 | 工具返回结果 | 高 | 经过验证的输出 |
| 中 | 文档/知识库 | 中 | 取决于版本和时效 |
| 低-中 | 用户输入 | 低-中 | 需要验证和签名确认 |
| 低 | dApp 页面内容 | 低 | 仅作为声称，不可信 |
| 最低 | Memory/历史记录 | 最低 | 不能用于授权判断 |

### Context Spec 设计

字段的完整元信息设计：

| 字段 | 说明 |
|------|------|
| field | 字段名 |
| value | 字段值 |
| source | 信息来源（用户/链上/文档/系统） |
| timestamp | 获取时间 |
| freshness | 是否实时/数据新鲜度 |
| trust_level | 可信度等级 |
| permission_scope | 能否用于判断身份/资产/权限 |
| chain_id | 属于哪条链 |
| block_number | 对应哪个区块 |
| verification_method | 如何验证的（签名/RPC/simulation） |
| can_be_used_as_fact | 能否当事实使用 |
| expiry | 什么时候过期 |
| notes | 额外说明 |

核心思想：每个上下文字段不只是内容，还要带来源、时效、权限、可信度四个维度的元信息。这样才能让 Agent 在 Web3 场景中做出安全的判断。
