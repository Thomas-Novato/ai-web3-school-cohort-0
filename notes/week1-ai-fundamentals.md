## 1. LLM — 大语言模型核心原理

### 1.1 是什么

LLM（Large Language Model）基于上下文进行**概率生成**：给定输入文本，预测最合理的下一个 token 序列。它不是数据库，而是推理引擎。

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| **Token** | 模型处理的基本单位，影响上下文容量和成本 |
| **Embedding** | 语义向量，用于搜索和 RAG，但不能判断正确性 |
| **Context Window** | 模型的"工作内存"，决定能"看到"多少信息 |
| **Transformer** | 核心架构，擅长找模式但不是数据库 |
| **Hallucination** | 模型自信编造不存在的信息，必须外部校验 |
| **Multimodal** | 多模态能力（文本/图像/代码），仍需结构化验证 |

### 1.3 关键认知

> **LLM 是推理层，不是真相源。**

它擅长：
- 语言理解、代码生成、推理

它不擅长：
- 精确事实记忆（容易产生幻觉）
- 确定性计算（可能算错）
- 跨会话状态保持（每次对话独立）

### 1.4 在 AI × Web3 中的位置

```
┌──────────────────────────────────────────────┐
│  数据层: RPC / 索引器 / 预言机                │
├──────────────────────────────────────────────┤
│  编排层: Prompt → Context → RAG → Agent      │
├──────────────────────────────────────────────┤
│  执行层: 工具调用 → 钱包 → 合约部署           │
├──────────────────────────────────────────────┤
│  安全层: Guard / Simulation / 权限管理         │
└──────────────────────────────────────────────┘
```

> **核心原则：** 链上数据是真相源，LLM 负责理解和推理。越靠近执行层，越需要外部验证。

## 2. Prompt — 提示词工程

### 2.1 三个控制层面

| 层面 | 作用 | 类比 |
|------|------|------|
| **上下文窗口** | 控制模型能看到多少信息 | 工作内存大小 |
| **系统指令** | 设置身份、语气、行为边界 | 岗位说明书 |
| **提示词（Prompt）** | 传递当前任务意图 | 具体任务指派 |

### 2.2 Prompt → Workflow → Agent 的边界

```
Prompt           ─ 让模型回答，决策在人
  ↓
Workflow        ─ 预定义流程，模型是一个节点，路径固定
  ↓
Agent           ─ 模型自主规划、动态调工具、跨轮管理状态
```

- Prompt 失控风险最低，可调试性最高
- Agent 失控风险最高，需要边界和人工确认
- 三者复杂度递增，**不是越复杂越好**

### 2.3 元 Prompt 思路

> 不需要追求"完美 prompt"。把想法告诉 AI，让 AI 帮你优化 prompt，效率更高。

```
我的想法 → AI 优化 → 更好的 prompt → 更好的结果
```

### 2.4 实践原则

- 重点不是格式，而是**清晰表达意图**
- **快速迭代**比理论分析更有效
- AI 本身可以成为优化 prompt 的工具

## 3. Context — 上下文与可信度管理

### 3.1 核心认知

> **Context 的质量 > Context 的数量。**

更大的上下文窗口不等于更可靠的判断：
- 模型对长上下文的**注意力分布不均匀**（lost in the middle）
- 大量无关/过期/冗余信息会**稀释关键信号**
- 更大的窗口 = 更大的风险面

### 3.2 可信度分层体系

| 层级 | 内容类型 | 可信度 | 说明 |
|:----:|----------|:------:|------|
| 🏅 最高 | **系统规则** | 最高 | 定义安全边界和行为约束 |
| ✅ 高 | **链上记录** | 高 | 客观事实，可验证 |
| ✅ 高 | **工具返回结果** | 高 | 经过验证的输出 |
| 📄 中 | **文档/知识库** | 中 | 取决于版本和时效 |
| 👤 低-中 | **用户输入** | 低-中 | 需要签名验证 |
| ⚠️ 低 | **dApp 页面内容** | 低 | 仅作为"声称"，不可信 |
| ❌ 最低 | **Memory/历史记录** | 最低 | 不能用于授权判断 |

### 3.3 场景示例

**Q: 用户说"这个地址是我的钱包"，模型能直接当事实吗？**

不能。需要回答：
1. 安全风险 — 可能是盗取/钓鱼地址
2. 身份冒用 — 地址可能属于别人
3. 人为失误 — 用户可能输错
4. 意图不明 — 用户可能只是想查询
5. **未验证** — 没有钱包签名确认

> **原则：** 涉及身份和资产时，必须经过当前会话的签名和授权确认。口头陈述不等于事实。

**Q: Memory 在 Web3 场景的最大风险是什么？**

Memory 不能保存或延续**授权**：
- 授权有**时效性** — 过去授权可能已被撤销
- 授权有**会话绑定** — 不能跨会话重用
- Memory 中的历史记录可能导致**过期授权幻觉**

### 3.4 Context Spec 元信息设计

每个上下文字段应携带**四维元信息**：

| 维度 | 字段 | 说明 |
|:----:|------|------|
| **来源** | `source` | 用户/链上/文档/系统 |
| **时效** | `timestamp`, `freshness`, `expiry` | 获取时间、新鲜度、过期时间 |
| **权限** | `permission_scope` | 能否用于判断身份/资产/权限 |
| **可信度** | `trust_level`, `verification_method` | 可信等级、验证方式（签名/RPC/Simulation） |

其他字段：`chain_id`, `block_number`, `can_be_used_as_fact`, `notes`

## 4. Agent — 智能体执行循环

### 4.1 Agent 的本质

> **Agent 不是更像人的 AI，而是一个有边界、有权限、有记录、有停止机制的任务执行系统。**

核心是**被约束的执行循环**，不是"自由意志"。

### 4.2 五大核心组件

| 组件 | 说明 | 类比 |
|------|------|------|
| **目标** | 要完成什么 | 任务委派 |
| **工具** | 能调用什么（API/合约/搜索） | 工具箱 |
| **状态** | 当前进度和中间结果 | 工作底稿 |
| **权限** | 什么能做、什么不能做 | 授权范围 |
| **停止条件** | 什么时候结束 | 完成/失败/超时 |

### 4.3 操作权限分级

| 操作类型 | 处理方式 |
|----------|---------|
| ✅ **只读操作** | 可以自动执行 |
| ⚠️ **高风险写入** | Policy 检查 → Simulation → 用户确认 |
| 📝 **状态管理** | 外置记录，不在模型上下文中（方便审计/追责） |

### 4.4 Agent 铁律（永不变的安全红线）

1. ❌ **Agent 绝不接触私钥 / 助记词**
2. ❌ **签名和转账必须有人工确认**
3. ❌ **Agent 不能绕过钱包授权**
4. ✅ 只读操作可自动执行
5. ✅ 高风险写入需 policy → simulation → 用户确认
6. ✅ 状态/日志应外置（方便恢复、审计、回滚）

### 4.5 核心技术组件

| 组件 | 说明 |
|------|------|
| **Tool Calling** | 模型输出结构化请求，框架执行后回传结果 |
| **MCP** | LLM 与外部工具的统一连接协议 |
| **Skills** | 可复用的高层指令集 |
| **Tracing** | 可视化 agent 执行链 |
| **Guardrails** | 输入/输出验证规则，不符合则中止 |
| **Handoff** | 子任务完成后移交控制权 |
| **错误恢复** | 重试、回退、人工介入 |

### 4.6 什么时候需要 Agent？

| 适合用 Agent | 更适合简单方案 |
|-------------|---------------|
| 目标开放式，无法提前写死步骤 | 一次性问答 → Prompt |
| 需要多工具协作 | 流程固定 → 脚本 |
| 中间结果决定下一步策略 | 高合规要求 → 人工审核 |
| 需跨会话记忆状态 | 数据确定性要求高 → 数据库查询 |

> **复杂度和风险越高，越要警惕过度 agent 化。**

### 4.7 AI × Web3 实践方向：DAO 提案研究 Agent

Agent 可以做的事：
- ✅ 分析提案内容
- ✅ 读取链上信息
- ✅ 生成交易草稿

Agent **不能**做的事：
- ❌ 绕过钱包授权
- ❌ 绕过用户确认
- ❌ 自动执行投票

**最小实践流程：** 资料读取 → 争议总结 → 风险标记 → 投票前检查清单
**不直接执行投票。**

| 原则 | 涉及章节 |
|------|:--------:|
| LLM 是推理层，不是真相源 | LLM |

| 任何输出必须外部验证 | Prompt, Agent |
| 签名/转账必须人工确认 | Agent |
| Agent 不接触私钥 | Agent |
| 状态外置不放在模型上下文中 | Agent |

## 6. 安全红线 Checklist

> 每次与 AI × Web3 交互前，过一遍这个清单。

- [ ] **信息来源核实**：这是链上数据/工具结果，还是用户自称/dApp 页面？
- [ ] **权限边界确认**：当前操作是只读还是写入？
- [ ] **人工确认到位**：涉及签名/转账/合约写入，有 human-in-the-loop 吗？
- [ ] **幻觉检查**：AI 输出的事实部分是否经过外部验证？
- [ ] **过期内容排查**：使用的文档/知识库是最新版本吗？
- [ ] **私钥隔离**：Agent 或其他 AI 工具是否接触到私钥/助记词？
- [ ] **状态外置**：关键状态和日志是否独立存储，不在模型上下文中？
- [ ] **Gas / 成本意识**：链上操作有成本意识吗？是否先在测试网验证？
